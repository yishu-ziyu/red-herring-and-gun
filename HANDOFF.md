## 当前已确认的核心方向

这个 Agent 不是普通搜索工具，也不是普通 fact-checker，而是面向知识工作者的 argument-checking agent。

核心目标：

> 帮用户把一个含混观点转化成可审查的论证结构，并根据用户身份与使用场景选择合适的证据路径。

## 方法论核心

### 语境化可核查分解

处理顺序不是：

> 原句 → 关键词 → 搜索

而是：

> 原句 → 语境 → 命题类型 → 可核查子命题 → 证据需求 → 查询关键词

关键词是最后生成的，不是最开始生成的。

## 产品核心差异

普通 AI 搜索：

> 用户问问题 → AI 搜索 → AI 总结

我们的产品：

> 用户提出观点 → AI 诊断论证结构 → 用户选择证据路径 → AI 按证据标准核查 → 回到原论证

一句话定位：

> 搜索引擎帮你找信息，我们帮你看清一个观点的结构。

## 用户范围

当前认为这些人都可能是用户：

- 本科生：课程论文、课堂讨论、读书笔记、理解观点
- 研究生：论文选题、文献综述、研究假设、证据判断
- 记者：事实核查、报道线索、时间线、多方证据
- 内容创作者：热点评论、视频脚本、公众号文章、观点表达
- 政策研究者：政策文本、制度背景、影响评估、风险判断
- 普通知识工作者：理解复杂观点，提高思考质量

这些人群仍可作为长期用户范围。黑客松 MVP 当前优先采用「信息真相猎人」叙事，面向需要快速判断热点传言、截图、链接和社交传播材料的人；学术/政策/写作场景保留为方法论扩展方向。

## 当前认为的高频痛点

1. 不会拆问题。
   - 用户输入的是宏大判断。
   - 普通 AI 容易直接发散搜索。
   - 我们要先判断说法由哪些命题构成。

2. 不会判断证据质量。
   - 用户拿到一堆资料，但不知道哪些证据强、哪些只是线索。

3. 容易被宏大判断带偏。
   - 例如“AI 会让文科生失去竞争力”“文科教育正在失去意义”。
   - 这类句子混杂概念、事实、比较、因果、预测、价值。

4. 写作质量差的根本原因往往不是表达能力，而是思考结构不清楚。

## 证据需求的最新理解

证据需求不是搜索关键词，而是：

> 为了判断一个子命题是否成立，用户需要哪一类证据，以及什么样的证据才算强。

更进一步，证据需求应该服务于用户身份和使用场景。

同一句话，不同用户的证据需求不同：

- 学术研究：论文、数据、变量、方法、反事实
- 课堂理解：概念、正反观点、基础资料、例子
- 内容创作：趋势、案例、冲突、反直觉、传播表达
- 新闻核查：原始来源、时间线、多方说法、反证
- 政策分析：政策文本、执行机制、受影响群体、风险

## 重要产品想法：路径化证据需求

Agent 不应该自己决定用户需要什么证据，而是先生成多条可能的论证/证据路径，让用户选择。

候选定义：

> Agent 根据用户身份和使用场景，生成多条可深入的论证路径；用户选择路径后，Agent 再展开对应的子命题、证据标准和核查报告。

## 白板式交互想法

用户输入一个观点后，页面像一张可拓展白板。

中心是原句，向外发散多条线：

- 概念线
- 事实线
- 比较线
- 因果线
- 预测线
- 价值线
- 策略线

用户点击某一条线后，系统沿着这条路径继续展开证据需求、子命题和证据结果。

这属于早期交互设想。当前 MVP 已从“路径卡片 + 右侧报告”演进为 Mission Control 办案台：左侧主控调度，中间当前阶段工作台，右侧或下方承载流式过程、证据矩阵和最终收束。

## 暂时认可的好拆分标准

好的拆分必须满足：

1. 不丢失原句争议重点。
2. 不把复杂判断粗暴拆成关键词。
3. 每个子命题只包含一种判断关系。
4. 每个子命题都能对应证据类型。
5. 子命题之间能还原原始论证链。
6. 能区分“可核查事实”和“不可直接核查的价值判断”。
7. 能主动指出概念偷换、因果跳跃、比较对象缺失。

用户暂时认可这些标准，但后续 MVP 体验后可能调整。

## 已确认：混合判断判定规则 v0.1

一个观点被判定为“混合判断”，当且仅当满足以下任一条件：

1. 同时出现两个以上命题类型：事实、概念、比较、因果、预测、价值、策略。
2. 关键概念没有定义，却承担了结论重量。
3. 出现因果词，但没有明确机制、时间顺序、替代解释。
4. 出现比较词，但没有明确比较对象、指标或时间范围。
5. 出现价值词，却没有说明评价标准。
6. 一个结论需要两种以上不同类型证据支持。

核心原则：

> 证据需求分裂，就是判断混合的标志。

## 初步架构想法：判断型 Sub Agent

用户提出：混合判断识别可以由一个专门的 sub agent 承担。这个 sub agent 应该有专门 prompt，背后也可以有专门的规则库/数据库来承接判断标准。

当前判断：这个方向是合理的。因为“判断层扫描”是产品链条中的独立认知动作，适合模块化：

- 输入：原句或一段观点
- 输出：命题类型、混合判断理由、触发规则、需要澄清的问题、下一步拆分建议
- 依据：混合判断判定规则、命题类型表、信号词表、证据需求类型表、示例库

MVP 阶段可以先用 prompt + 规则表实现，不必一开始做复杂数据库。后续可沉淀为可维护的规则库与示例库。

## 已确认：Claim Decomposer 子命题拆分原则 v0.1

用户认可当前定义：

> 一个好的子命题，是原观点中的一个最小可审查判断。它只表达一种判断关系，并且可以明确说出需要什么证据来支持或反驳。

好的子命题应满足：

1. **不百科化**：不把观点粗暴拆成“XX 是什么”这类孤立概念，除非该概念确实承担论证重量。
2. **不混合判断**：每个子命题只处理事实、概念、比较、因果、预测、价值、策略中的一种主判断。
3. **保留原争议**：拆完后仍然能看出它在回应原句，而不是散乱问题。
4. **可核查**：子命题必须能说清楚支持或反驳它需要什么证据。
5. **有论证位置**：标明它是前提、机制、比较、反证，还是结论边界。

重要提醒：用户意识到自己过去在辩论或分析中也常做“百科化拆解”；这个产品应帮助用户避免把论证问题退化为百科问题。

核心原则：

> 拆解不是把句子拆成名词解释，而是把论证拆成可审查判断。

## 已确认：Evidence Router 证据适配原则 v0.1

用户对初始“证据路由表”做了关键修正：证据强弱不能只按来源权威判断，而要看证据是否适配当前子命题。

核心定义：

> Evidence Router 不是搜索模块，而是证据适配模块。它根据子命题类型和使用场景，决定什么证据有资格支持该命题，什么证据只能作为线索，什么证据不能使用。

### 证据强度四维度

证据强不强，不只看来源，还要看四个维度：

| 维度 | 问的问题 |
| --- | --- |
| 相关性 | 这份材料是否正面回答这个子命题？ |
| 可追溯性 | 能否回到原文、原始数据、原始记录？ |
| 方法适配 | 这类证据是否适合处理这种命题？ |
| 语境适配 | 这份证据的时间、地区、人群、学科语境是否匹配？ |

示例判断：

- 官方文件对“政策是否出台”是强证据；对“政策是否有效”只是背景材料。
- 行业报告对“企业如何描述趋势”有价值；对“AI 是否导致岗位减少”通常不够。
- RCT 对某个干预的内部效度强；但未必能外推到另一个国家、行业或群体。
- 评论文章对事实核查通常弱；但对“某种社会观点如何被表达”可能是可用材料。
- 访谈对个体经验强；对总体趋势弱。
- 统计数据对趋势和比较强；对概念含义和价值判断弱。

### 修正版证据路由表

| 子命题类型 | 核心问题 | 优先证据 | 禁用或降权证据 |
| --- | --- | --- | --- |
| 概念 | 这个词在此处如何定义，能否操作化？ | 原文语境、正式定义、学术定义、用法比较 | 脱离语境的百科解释、随意引申 |
| 事件事实 | 这件事是否发生？ | 官方文件、原始公告、当事方记录、时间线 | 二手转述、截图、无来源文章 |
| 数量事实 | 某个数量或趋势是否存在？ | 同口径数据、统计说明、数据库、时间序列 | 无口径数据、孤立数字、转引数据 |
| 文本归属事实 | 谁说了什么，原文是什么？ | 原文、录音、视频、逐字稿、上下文 | 截断引用、二创内容、转述 |
| 比较 | A 是否比 B 更高/更低/更严重？ | 同一口径、同一时间范围、同一指标的数据 | 不同口径硬比、个案代替总体 |
| 因果 | X 是否导致 Y？ | 识别设计、机制链、反事实、替代解释、过程证据 | 同期发生、单个案例、情绪化评论 |
| 预测 | 未来是否可能发生 Y？ | 趋势、模型假设、多情景、不确定性、历史表现 | 单一路径外推、热点判断、专家口号 |
| 价值 | 在什么标准下它是好/坏/有意义？ | 评价标准、理论论证、事实支撑、反方框架 | 单一指标替代全部价值、情绪表达 |
| 策略 | 对谁，在什么目标下，该怎么做？ | 目标函数、方案比较、成本收益、风险、适用条件 | “大家都该做”、成功学案例、趋势焦虑 |

### 事实命题细分

“事实命题”必须拆成至少三类：

| 事实类型 | 例子 | 优先证据 |
| --- | --- | --- |
| 事件事实 | 某政策是否在 2023 年出台 | 官方文件、公告、原始记录 |
| 数量事实 | 某类岗位数量是否下降 | 统计数据、招聘数据库、口径说明 |
| 文本归属事实 | 某人是否说过某句话 | 原文、录音、视频、逐字稿、上下文 |

### 概念命题新增关键词：操作化

概念命题不能只问“这个词是什么意思”，还要问：

> 这个词在当前论证里能不能变成可观察、可比较、可讨论的对象？

例如“竞争力”可以操作化为就业率、起薪、职业稳定性、岗位选择权、跨行业迁移能力、不可替代能力、社会声望、长期收入增长等。不同操作化方式可能改变原论证结论。

概念命题目标应改为：

> 这个概念在当前语境中应如何定义？能否被操作化？不同定义会不会改变原论证？

### 因果命题证据层级

因果证据不应只按经济学强识别理解，应分三档：

| 因果证据层级 | 适用场景 |
| --- | --- |
| 强因果证据 | RCT、DID、IV、自然实验、清晰识别设计 |
| 机制型证据 | 时间顺序、机制链、访谈、过程追踪、案例比较 |
| 弱因果线索 | 同期变化、媒体叙述、单个案例、相关性描述 |

当无法获得严格识别时，至少要检查时间顺序、机制、替代解释和反例，阻止“同时发生 = 因果关系”。

### 预测命题判定标准

“权威机构预测”不能天然算强证据。预测强弱取决于：

- 假设是否公开；
- 模型或推理路径是否清楚；
- 变量是否合理；
- 是否给出不确定性；
- 过去类似预测的表现如何；
- 是否有多情景而非单一路径。

### 价值命题与策略命题

价值命题和策略命题不是单纯“证据命题”，而是“证据 + 标准”的混合命题。

| 类型 | 先问什么 | 再找什么 |
| --- | --- | --- |
| 价值命题 | 评价标准是什么？ | 哪些事实支持或削弱这个标准下的判断 |
| 策略命题 | 目标函数是什么？ | 成本、收益、风险、替代方案、适用条件 |

这两类命题必须明确标记：不能直接判真伪，只能在某个标准下评估。

### 用户身份与使用场景

用户身份不能改变事实本身，只能改变“到什么程度算够用”。它主要改变三件事：

1. 证据门槛；
2. 证据格式；
3. 证据优先级。

应增加字段：`use_context`，表示这次核查是为了什么。

候选使用场景：

| 使用场景 | 证据标准 |
| --- | --- |
| 快速理解 | 可信来源 + 基本解释 + 不确定性提示 |
| 课程作业 | 可引用来源 + 概念清楚 + 基本反方材料 |
| 学术研究 | 原始数据/文献 + 方法说明 + 反事实/反证 |
| 新闻核查 | 原始记录 + 多方交叉验证 + 时间线 |
| 政策/商业决策 | 数据 + 机制 + 成本收益 + 风险 |
| 个人行动 | 目标匹配 + 方案比较 + 可执行性 |

Evidence Router 的正式逻辑：

```text
子命题
→ 命题类型
→ 使用场景
→ 证据需求
→ 证据强度标准
→ 不适用证据提醒
→ 最低可接受输出
```

### 最低可接受输出规则

每个子命题都应有“没查够时不能乱说”的规则：

- 概念命题没找到原文语境时，只能说“可能有几种定义”，不能替用户断定作者含义。
- 事实命题没找到原始来源时，只能说“有二手来源提及”，不能说“已确认”。
- 比较命题没有同口径数据时，只能说“现有材料不能支持比较”。
- 因果命题没有反事实或机制时，只能说“存在相关或时间相邻”，不能说“导致”。
- 预测命题没有假设和不确定性时，只能说“这是判断，不是可靠预测”。
- 价值命题没有评价标准时，只能说“该判断标准不明”。
- 策略命题没有目标函数时，只能说“无法判断是否应该”。

这条规则是产品质量的关键：事实核查系统最重要的能力之一，不是给答案，而是知道什么时候不能给确定答案。

核心原则：

> 证据不是越多越好，而是要和命题类型、使用场景、语境边界匹配。

> 用户身份不能改变事实本身，只能改变证据门槛、格式和优先级。

> Router 必须阻止用户和 AI 用错证据。

## 已确认：Evidence Searcher 检索计划原则 v0.1

用户确认：Evidence Searcher 不应该直接联网搜索，而应该先生成 search plan，再按计划检索。

原因：搜索引擎按相关性、热度、SEO 和新近程度排序，但这些排序标准不等于证据质量。如果 Agent 直接搜，很容易把“最容易搜到的东西”误当成“最该找的证据”。

核心定义：

> Evidence Searcher 是证据获取模块。它不判断结论真假，而是根据 Evidence Router 给出的证据需求，生成检索计划、执行检索、记录检索路径，并把候选材料交给 Evidence Grader。

### 模块边界

| 模块 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| Evidence Router | 判断这个子命题理论上需要什么证据 | 不实际搜索 |
| Evidence Searcher | 决定去哪里找、用什么词找、找哪些正反材料 | 不判定证据强弱 |
| Evidence Grader | 判断找到的材料是否足够、是否适配命题 | 不负责扩大搜索范围，除非要求返工 |

### Searcher 核心输出

Searcher 的核心输出不是“搜索结果列表”，而是结构化 search plan。每个子命题至少生成：

| 字段 | 含义 |
| --- | --- |
| evidence_need | 这个子命题需要什么证据 |
| source_classes | 应该优先找哪类来源 |
| search_locations | 应该去哪些地方找 |
| query_sets | 具体搜索词组 |
| counter_queries | 专门找反证、替代解释、限制条件的搜索词 |
| stop_rule | 找到什么程度可以停止，没找到什么必须标记证据不足 |

核心原则：

> Searcher 的搜索词不是从原句直接提取，而是从证据需求生成。

### 检索模式

| 检索模式 | 适用情况 | 例子 |
| --- | --- | --- |
| 已知来源检索 | 找某个原文、政策、讲话、论文 | 某政策原文、某人原话 |
| 探索性检索 | 先了解问题有哪些说法和关键词 | 新概念、新热点 |
| 数据检索 | 找统计数据、数据库、长期趋势 | 就业率、岗位数量、工资 |
| 学术检索 | 找论文、理论、因果识别、综述 | AI exposure、DID、IV |
| 反证检索 | 主动找相反证据和替代解释 | “不是 AI 导致，而是经济周期” |
| 语境检索 | 找原文上下文、作者语境、概念来源 | 概念命题、文本归属事实 |
| 引用链检索 | 从二手材料回到原始来源 | 媒体引用报告、报告引用数据 |

### 五轮检索顺序

1. **定位语汇**：确认问题在不同语境里的常用表达，防止关键词太中文化、太口语化。
2. **找高优先级来源**：根据 Router 要求，优先找最适合命题的材料。
3. **找反证和替代解释**：尤其是比较、因果、预测、策略命题，没有反证搜索就不合格。
4. **回到原始来源**：如果找到的是二手材料，尝试回溯其引用的论文、数据、公告、原文。
5. **生成 evidence gap**：明确没找到什么，不假装已经完成核查。

### Searcher schema 草案

```json
{

    "劳动力市场报告",
    "企业案例",

  "query_sets": {
    "academic": [
      "generative AI occupation exposure writing jobs",
```

### 不同命题类型的 Searcher 规则

| 子命题类型 | Searcher 应该优先找 | 必须避免 |
| --- | --- | --- |
| 概念 | 原文、定义、学术用法、不同语境比较、操作化方式 | 只找百科定义 |
| 事件事实 | 原始公告、官方文件、当事方记录、时间线 | 只找转述文章 |
| 数量事实 | 数据库、统计表、口径说明、时间序列 | 孤立数字、无来源图表 |
| 文本归属事实 | 原文、逐字稿、录音视频、上下文 | 截图、短视频切条、二手转述 |
| 比较 | 同一数据源、同一指标、同一时间范围 | 不同口径硬比 |
| 因果 | 因果识别、机制、过程证据、替代解释、反事实 | 同期变化直接写成因果 |
| 预测 | 模型假设、多情景、历史趋势、不确定性 | 单一专家判断 |
| 价值 | 评价标准、理论框架、正反论证、相关事实 | 用一个数据替代全部价值判断 |
| 策略 | 目标函数、备选方案、成本收益、风险、适用人群 | 趋势焦虑、成功学案例 |

### 来源独立性与引用链

Searcher 需要记录 source_lineage。很多资料看起来有 10 个来源，其实都引用同一份报告。如果 10 篇文章都来自同一个原始数据，不能算 10 份独立证据，只能算 1 条证据链的多次传播。

示例：

```text
source_lineage:
- A 媒体文章引用 B 报告
- B 报告引用 C 数据
- C 数据是否可打开、是否有口径说明
```

### 防止 query drift

Searcher 必须防止检索漂移。原子命题是“AI 是否导致初级内容岗位减少”，搜索过程中不能漂移成：

- AI 是否影响就业；
- AI 是否替代人类；
- 文科生是否应该学 AI；
- 内容行业是否变差。

这些都相关，但不是同一个命题。每条候选材料都要记录它匹配哪个 evidence_need。匹配不上不能进入主证据区，只能放入背景材料。

### 候选证据卡 schema

```json
{
  "candidate_id": "E12",
  "subclaim_id": "C3",
  "source_title": "...",
  "source_type": "学术论文/报告/新闻/官方数据",
  "matched_evidence_need": "岗位需求时间序列",
  "why_collected": "它包含 2021-2024 年相关岗位招聘变化",
  "original_source_available": true,
  "possible_limitations": [
    "只覆盖美国市场",
    "岗位分类不完全等同于内容运营",
    "没有直接识别 AI 因果效应"
  ],
  "status": "交给 Evidence Grader"
}
```

Searcher 不要写“这是强证据”。它只能写“为什么把它收进候选材料”。强弱判断交给 Grader。

### MVP 分阶段

1. 第一版：Searcher 生成 search plan 和检索词，用户或系统把材料输入进来。
2. 第二版：接搜索 API 或浏览器检索，把候选链接抓回来。
3. 第三版：做来源回溯、反证检索、引用链整理。

黑客松版本第一版和第二版足够。重点不是做大爬虫，而是展示：同一句话，系统会因为命题类型不同而生成不同检索计划。

前端展示三块：

```text
1. 我需要什么证据
2. 我准备去哪里找
3. 哪些材料不能支持这个结论
```

### Searcher 最低可接受输出

每个子命题至少给出：

1. 优先来源类型；
2. 3-5 组检索词；
3. 反证检索词；
4. 不适用证据提醒；
5. 当前证据缺口。

如果这五项没有生成，就不应该进入 Evidence Grader。

核心原则：

> Evidence Searcher 不是把问题丢给搜索引擎，而是把证据需求翻译成检索计划。它要找的不只是支持材料，还包括原始来源、反证、替代解释和证据缺口。

Searcher 最容易出问题的地方：

1. 找到了相关材料，但不是适配证据。
2. 只找支持材料，没有找反证。
3. 找到了二手材料，没有回到原始来源。
4. 搜索过程中命题漂移，最后回答的不是原问题。

## 已确认：Evidence Grader 证据分级原则 v0.1

用户确认：Grader 可以用“相关性、可追溯性、方法适配、语境适配”四维度作为基础，但还必须增加证据方向、推理权限、独立性和子命题整体充分性。

核心定义：

> Evidence Grader 判断候选材料在当前子命题中能合法承担什么角色：支持、反驳、限定、背景、线索或不可用，并明确它能推出什么、不能推出什么。

关键原则：

> 不要问：这份材料强不强？
> 要问：这份材料对这个子命题，在这个使用场景下，能承担什么证据角色？

评分对象不是 source，而是 source-claim fit，即“材料—子命题”关系。

### 单条候选材料评分字段

| 字段 | 含义 |
| --- | --- |
| evidence_role | 支持 / 反驳 / 限定 / 背景 / 线索 / 不可用 |
| matched_need | 它对应 Router 提出的哪一项证据需求 |
| relevance | 是否正面回答当前子命题 |
| traceability | 是否能追到原文、原始数据或原始记录 |
| method_fit | 证据方法是否适合这个命题类型 |
| context_fit | 时间、地点、人群、行业、学科语境是否匹配 |
| independence | 是否是独立来源，还是转引同一个源头 |
| inference_allowed | 可以推出什么 |
| inference_blocked | 不能推出什么 |
| usage_level | 主证据 / 辅助证据 / 背景材料 / 仅作线索 / 不可用 |
| grader_note | 为什么这样判 |

其中 `evidence_role`、`inference_allowed`、`inference_blocked` 必须有。没有这三个字段，Grader 容易把“看起来相关”误判为“可以支持”。

### 示例：AI 是否导致初级内容岗位减少

候选材料 A：一篇研究发现写作类职业对大语言模型暴露度较高。

Grader 应判断：

```json
{
  "evidence_role": "辅助证据",
  "matched_need": "机制可能性 / 职业暴露度",
  "relevance": "中",
  "traceability": "高",
  "method_fit": "中",
  "context_fit": "视研究对象而定",
  "inference_allowed": [
    "可以说明写作类职业可能受到生成式 AI 影响",
    "可以作为机制背景"
  ],
  "inference_blocked": [
    "不能推出初级内容岗位已经减少",
    "不能推出岗位减少由 AI 导致",
    "不能推出文科生整体竞争力下降"
  ],
  "usage_level": "辅助证据"
}
```

候选材料 B：某公司宣布因为 AI 工具减少内容团队招聘。

Grader 应判断：

```json
{
  "evidence_role": "线索 / 案例材料",
  "matched_need": "企业采用 AI 后岗位变化",
  "relevance": "中",
  "traceability": "中",
  "method_fit": "低",
  "context_fit": "低到中",
  "inference_allowed": [
    "可以说明存在个别企业用 AI 替代或减少内容岗位的案例"
  ],
  "inference_blocked": [
    "不能推出行业总体岗位减少",
    "不能推出因果关系普遍成立",
    "不能排除企业降本、业务收缩等其他原因"
  ],
  "usage_level": "仅作线索"
}
```

候选材料 C：招聘数据库显示 2021-2024 年内容写作类初级岗位数量下降，并说明岗位分类和数据口径。

Grader 应判断：

```json
{
  "evidence_role": "支持",
  "matched_need": "岗位数量或招聘需求时间序列",
  "relevance": "高",
  "traceability": "高",
  "method_fit": "高",
  "context_fit": "取决于地区和岗位分类",
  "inference_allowed": [
    "可以支持某一口径下初级内容岗位招聘需求下降"
  ],
  "inference_blocked": [
    "不能单独推出 AI 是下降原因",
    "不能排除宏观经济、行业周期、平台变化等替代解释"
  ],
  "usage_level": "主证据：仅用于数量事实部分"
}
```

同一份材料对不同子命题的用途不同：

- 招聘数据对“岗位是否减少”可能是主证据。
- 对“AI 是否导致岗位减少”只是部分证据。
- 对“文科生是否应该学 AI”只能作为背景材料之一。
- 对“文科教育是否失去意义”不能直接作为主证据。

### Grader 两层结构

第一层：单条证据评分。

| 维度 | 高 | 中 | 低 |
| --- | --- | --- | --- |
| 相关性 | 直接回答子命题 | 回答相关部分 | 只提供背景或偏题 |
| 可追溯性 | 可回到原文、原始数据、原始记录 | 有明确出处但非原始来源 | 来源不明、转述、截图 |
| 方法适配 | 方法正好适合命题类型 | 方法部分适合 | 方法不能支撑该判断 |
| 语境适配 | 时间、地点、人群、对象匹配 | 部分匹配 | 语境差异很大 |
| 独立性 | 独立来源或独立数据 | 部分转引 | 完全依赖同一源头 |

第二层：子命题整体充分性。

| 判断 | 含义 |
| --- | --- |
| sufficiently_supported | 关键证据需求已满足，反证和限制已处理 |
| partially_supported | 支持某一部分，但不能支持完整命题 |
| contradicted | 主要证据反而削弱或反驳该命题 |
| inconclusive | 有材料，但不足以支持或反驳 |
| not_checkable_yet | 当前材料无法核查，需更多原始来源或数据 |
| misframed | 子命题本身表述有问题，需要重写 |

### 子命题整体判断示例

对“AI 导致初级内容岗位减少”，可能的结论是：

```text
数量事实部分：部分支持。某些数据表明相关岗位招聘需求下降。
机制部分：有辅助证据。写作类任务对生成式 AI 暴露度较高。
因果部分：证据不足。当前材料不能排除宏观经济、行业收缩和企业降本等替代解释。
最终判断：不能写成“AI 导致岗位减少”；更严谨的写法是“生成式 AI 可能是影响初级内容岗位需求的因素之一，但现有材料不足以确认其为主要原因。”
```

### 命题类型致命缺陷规则

这些规则优先于综合评分：

| 命题类型 | 致命缺陷 |
| --- | --- |
| 概念 | 没有原文语境，却断定作者含义 |
| 事件事实 | 没有原始记录，却说“已确认发生” |
| 数量事实 | 没有口径说明，却比较数字 |
| 文本归属事实 | 没有原文或完整上下文，却说“某人说过” |
| 比较 | A 和 B 数据口径不同，却直接比较 |
| 因果 | 只有同期变化，却说 X 导致 Y |
| 预测 | 没有假设和不确定性，却给确定结论 |
| 价值 | 没有评价标准，却判定好坏 |
| 策略 | 没有目标函数，却说“应该做” |

### Grader schema 草案

```json
{
  "subclaim_id": "C3",
  "candidate_id": "E12",
  "matched_evidence_need": "岗位需求时间序列",
  "evidence_role": "support",
  "scores": {
    "relevance": "high",
    "traceability": "high",
    "method_fit": "high_for_quantity_fact_low_for_causality",
    "context_fit": "medium",
    "independence": "medium"
  },
  "usage_level": "main_evidence_for_quantity_fact",
  "inference_allowed": [
    "在该数据口径下，初级内容岗位招聘需求下降"
  ],
  "inference_blocked": [
    "不能单独推出 AI 导致下降",
    "不能外推到所有文科岗位",
    "不能代表所有地区和平台"
  ],
  "limitations": [
    "只覆盖某一市场",
    "岗位分类可能不完整",
    "缺少 AI 采用变量"
  ],
  "evidence_gap": [
    "仍需替代解释材料",
```

不要让 Grader 输出笼统总分。总分会制造假精确。高/中/低加文字解释更适合这个产品。

### 与 Report Composer 的接口

Report Composer 不能随意引用所有候选材料，只能按 Grader 的 `usage_level` 使用。

| Grader 用途 | Report Composer 怎么用 |
| --- | --- |
| 主证据 | 可放入核心证据链 |
| 辅助证据 | 可用于补充机制、背景、趋势 |
| 背景材料 | 只能解释语境，不能支撑结论 |
| 仅作线索 | 只能提示“有待核实” |
| 不可用 | 不进入报告正文，最多放入排除记录 |
| 反证 | 必须进入不确定性或反方证据部分 |

核心原则：

> Evidence Grader 的任务不是给资料贴“强/弱”标签，而是判断资料在当前子命题中被允许怎样使用。它必须输出证据角色、适配评分、可推出内容、不可推出内容和剩余缺口。

Grader 需要补上的关键能力：证据方向、独立性、推理权限，以及子命题整体充分性。

## 已确认：Report Composer 报告合成原则 v0.1

用户修正：报告最前面应该有“结论卡”，让用户先知道当前能说到什么程度；“不能推出什么”不能只放在报告末尾，而应同时出现在每个子命题旁边，因为过度推断通常发生在某条证据被错误使用时。

核心定义：

> Report Composer 是受 Evidence Grader 约束的合成模块。它只能把已获得许可的证据角色、推理范围和证据缺口，组织成用户可读、可引用、可继续调查的报告。

Composer 的输入不是原始搜索结果，而是 Grader 处理后的材料：

```text
subclaim status
evidence role
usage level
inference_allowed
inference_blocked
limitations
evidence_gap
```

Composer 不应直接看一篇文章后自由发挥，只能从 Grader 输出中取东西。

### Composer 权限表

| Grader 结果 | Composer 允许怎么写 |
| --- | --- |
| 主证据 | 可进入核心证据链 |
| 辅助证据 | 只能支持机制、背景、趋势解释 |
| 背景材料 | 只能解释语境，不能支撑结论 |
| 仅作线索 | 只能写“有待核实”，不能写成事实 |
| 不可用 | 不进入正文证据链 |
| 反证 | 必须进入限制、反方证据或结论边界 |
| 因果不足 | 不能使用“导致”“造成”“决定”等因果表达 |
| 价值标准不明 | 不能替用户判定“好/坏/有意义/无意义” |
| 目标函数不明 | 不能写“应该做/不应该做” |

### 语言权限表

| 表达 | 允许条件 |
| --- | --- |
| “已发生” | 有原始记录或可靠事件证据 |
| “某数据下降/上升” | 有数据来源、统计定义、时间范围 |
| “A 高于 B” | A 和 B 可比，指标一致 |
| “可能相关” | 有相关材料，但不能证明因果 |
| “可能是因素之一” | 有机制或部分证据，但不能排除其他解释 |
| “导致” | 有足够因果证据，至少处理时间顺序、机制和替代解释 |
| “应该” | 已明确目标、对象、成本收益和替代方案 |
| “有意义/无意义” | 已明确评价标准和价值框架 |
| “未来会” | 有模型、假设、趋势和不确定性说明 |

这张表用于防止报告最后把“可能”“相关”“趋势”“案例”都写成确定结论。

### 报告结构

一个好的核查报告建议结构：

```text
0. 结论卡
1. 原句诊断
2. 子命题地图
3. 证据状态表
4. 证据链说明
5. 不能推出什么
6. 更严谨改写
7. 下一步需要的证据
8. 附录：证据卡与排除记录
```

### 0. 结论卡

结论卡要非常短，直接告诉用户当前判断。

模板：

```text
原句：
“AI 导致初级内容岗位减少。”

当前判断：
原句过强。现有材料可以支持“生成式 AI 可能影响写作和内容类岗位的任务结构”，也可能支持“部分地区或平台的初级内容岗位招聘需求下降”。但如果缺少同一统计定义下的岗位时间序列、AI 采用证据和替代解释检验，不能写成“AI 导致岗位减少”。

可说到：
AI 可能是影响初级内容岗位需求的因素之一。

不能说：
不能确认 AI 是主要原因。
不能把个别企业案例外推到整个行业。
不能把“职业暴露度高”直接等同于“岗位已经减少”。
```

### 1. 原句诊断

指出原句混合了哪些判断，以及为什么不能直接查一个关键词判断真假。

示例：

```text
这句话至少混合了三类判断：

1. 数量事实：初级内容岗位是否减少？
2. 因果判断：岗位减少是否由 AI 导致？
3. 范围判断：这个变化发生在哪些地区、行业、平台和时间段？

其中，“岗位是否减少”可以用招聘数据或劳动市场数据核查；“是否由 AI 导致”需要机制、时间顺序、反事实或替代解释；“初级内容岗位”需要先定义岗位范围。
```

### 2. 子命题地图

把原句变成可审查对象。

| ID | 子命题 | 类型 | 当前需要的证据 |
| --- | --- | --- | --- |
| C1 | “初级内容岗位”指哪些岗位？ | 概念 | 岗位定义、招聘分类、行业语境 |
| C2 | 某一范围内初级内容岗位招聘需求是否下降？ | 数量事实 | 同一统计定义下的招聘时间序列 |
| C3 | 生成式 AI 是否能替代这些岗位中的部分任务？ | 机制/事实 | 任务暴露度研究、企业流程材料 |
| C4 | 岗位下降是否主要由 AI 导致？ | 因果 | 时间顺序、机制、替代解释、对照证据 |
| C5 | 是否存在反向证据，例如 AI 创造新内容岗位？ | 反证 | 新岗位数据、企业招聘变化、技能需求变化 |

子命题地图不应太长，目的是让用户看清论证骨架。

### 3. 证据状态表

状态分类：

```text
支持
部分支持
反驳
限定支持
证据不足
不可核查
原命题需要改写
```

“限定支持”很重要。很多文科判断不是全错，而是说得太大。

示例：

| 子命题 | 证据状态 | 可用证据 | 不能推出 |
| --- | --- | --- | --- |
| C1 岗位定义 | 部分明确 | 招聘分类、行业报告中的岗位描述 | 不能把所有文科岗位都归入内容岗位 |
| C2 岗位是否下降 | 视数据而定 | 招聘数据、平台岗位数量变化 | 没有同一统计定义时不能比较 |
| C3 AI 是否替代部分任务 | 较可能支持 | 职业暴露度研究、企业使用案例 | 不能推出岗位已经减少 |
| C4 AI 是否导致岗位下降 | 证据不足 | 需要因果研究或机制 + 替代解释 | 不能用同期变化写成因果 |
| C5 反证 | 必须检查 | AI 相关新岗位、技能需求变化 | 不能只找支持材料 |

### 4. 证据链说明

不是堆资料，而是解释证据之间的关系。

示例：

```text
当前证据链最多能支持到：

任务层面：
写作、摘要、基础编辑、内容生成等任务确实是生成式 AI 能力较强的区域。

岗位层面：
如果招聘数据表明相关岗位减少，可以支持“某一范围内岗位需求下降”。

因果层面：
只有在证明 AI 采用早于岗位变化，并且处理了宏观经济、行业收缩、企业降本等替代解释之后，才可以进一步讨论“AI 导致岗位减少”。

因此，目前不能直接从“AI 能做内容任务”跳到“AI 导致初级内容岗位减少”。
```

### 5. 不能推出什么

这一节要明确、具体，不能写成泛泛免责声明。

示例：

```text
当前材料不能推出：

1. 不能从“AI 能完成写作任务”推出“内容岗位已经减少”。
2. 不能从“某些企业减少内容招聘”推出“整个行业减少”。
3. 不能从“岗位减少发生在生成式 AI 普及之后”推出“岗位减少由 AI 导致”。
4. 不能从“初级内容岗位受影响”推出“文科生整体竞争力下降”。
5. 不能把“岗位被替代”与“任务被重组”混为一谈。
```

### 6. 更严谨改写

这是 Report Composer 最有价值的输出之一。它把原句改成证据允许的版本。

应允许多个版本：

| 版本 | 用途 |
| --- | --- |
| 谨慎版 | 学术、研究报告、课程论文 |
| 通俗版 | 演讲、展示、短文 |
| 反驳版 | 用于指出原句哪里过强 |
| 待查版 | 用于继续调查 |

示例：

```text
生成式 AI 可能正在改变初级内容岗位的任务结构。现有证据如果仅能说明写作类任务具有较高自动化暴露度，或部分企业减少相关招聘，那么还不足以确认“AI 导致岗位减少”。更谨慎的说法是：AI 可能是影响初级内容岗位需求的因素之一，但仍需结合招聘数据、企业采用情况和宏观经济等替代解释进一步判断。
```

### 7. 下一步需要的证据

不能泛泛写“需要更多研究”，要具体写缺什么，并按优先级排列。

示例：

```text
要把这个判断从“可能相关”推进到“较有因果支持”，至少需要：

1. 同一统计定义下的初级内容岗位招聘时间序列。
2. 企业或行业采用生成式 AI 的时间证据。
3. 能排除宏观经济、广告市场收缩、平台变化、企业降本等替代解释的材料。
4. 关于内容岗位任务结构变化的机制证据。
5. 反向证据：AI 是否创造了新的内容岗位或提高了相关岗位需求。
```

### 8. 附录：证据卡与排除记录

面向需要严谨性的用户，可默认折叠但系统保留。

证据卡字段：

- 材料名称；
- 来源类型；
- 对应子命题；
- Grader 允许用途；
- 可以推出；
- 不能推出；
- 限制。

排除记录字段：

- 哪些材料没有进入报告；
- 为什么不能用；
- 是来源不明、方法不适配、语境不匹配，还是只能作为线索。

排除过程本身就是专业性。

### 合成强度规则

> 结论强度不得超过关键证据链中最弱的一环。

例如“AI 导致初级内容岗位减少”至少需要三段成立：

1. 初级内容岗位确实减少；
2. AI 能影响这些岗位的任务或用工需求；
3. 岗位减少主要或部分由 AI 引起，而不是其他因素。

如果第 1 段成立，第 2 段成立，第 3 段证据不足，最终报告不能写“AI 导致岗位减少”，只能写“AI 可能与岗位变化有关，但因果证据不足”。

### Composer 禁止动作

| 禁止动作 | 例子 |
| --- | --- |
| 把线索写成事实 | “有企业案例”写成“行业已经发生” |
| 把相关写成因果 | “同期出现”写成“导致” |
| 把局部写成整体 | “内容岗位”写成“文科生就业” |
| 把任务替代写成岗位消失 | “AI 能写文案”写成“文案岗位会消失” |
| 把价值判断伪装成事实 | “就业率下降”写成“文科教育无意义” |
| 把预测写成确定未来 | “可能继续变化”写成“未来必然恶化” |
| 把背景材料放进核心结论 | 用评论文章支撑因果判断 |

## 当前已确认的产品定位

用户确认：这个产品优先解决的是**思考质量**，不是写作质量。

写作质量只是表层结果；真正的问题是用户不会拆解观点、不会判断证据、容易被宏大判断带偏。产品应该帮助用户提升对观点、命题、证据和论证结构的理解能力。

## 2026-05-18 MVP 初版反馈：当前不是产品

用户看完 MVP 初版后的第一反馈：当前页面“根本不像一个产品”，只是把讨论过的方法论堆在一起，忽略了用户体验。

关键教训：

> 方法论完整不等于产品可用。产品不应该把内部模块全展示给用户，而应该把用户带进一个自然、低负担、有反馈感的使用流程。

当前初版问题：

1. 像内部方法论仪表盘，不像用户产品。
2. 信息密度过高，用户一上来被 Scanner / Router / Searcher / Grader / Composer 等内部概念压住。
3. 缺少用户动作：用户没有感觉自己在“选择路径、推进调查、获得洞见”。
4. 缺少产品节奏：没有从输入、困惑、分叉、证据选择、结论收束的体验曲线。
5. 没有把“把观点改写到证据允许的强度”转化成一个可感知的交互瞬间。

下一步必须先重做 UX，而不是继续堆模块。内部模块仍然存在，但应该隐藏在产品流程之后。

## 2026-05-18 MVP 二版反馈：线性流程仍然偏离最初构想

用户看完第二版后的反馈：虽然不再像方法论陈列页，但仍然是线性的、古板的流程页，没有回到最初的核心构想——无限扩张白板，让思路自然延伸。

关键问题：

1. 当前页面仍然是流式流程，不是白板式思考空间。
2. 过度依赖“左右对照 / 可以说 vs 不能说”的静态阅读，要求用户读表，而不是看思路生长。
3. 没有体现真正的 AI reasoning 过程，只是把预置数据和简单代码逻辑展示出来。
4. 没有达到 Agent 标准：理论上 Agent 应该体现思考过程、路径选择、展开、回溯、证据判断和结论收束。
5. UI 没有合理呈现 reasoning：用户看不到系统为什么沿某条路径推进、为什么分叉、为什么停下、为什么回收结论。

新的核心方向：

> 产品应该是“可视化 reasoning 白板”，不是“线性报告流程”。

下一步应该重新设计为：中心观点节点 → 判断层节点 → 子命题节点 → 证据需求节点 → 候选证据节点 → 推理许可节点 → 改写结论节点。用户点击任一节点都能沿该路径继续展开，形成可增长的论证图谱。

Agent 能力必须显性化：不能只靠静态预置数据，而要有“reasoning trace / agent steps / path expansion”的表现形式，即使 MVP 阶段先用预置 reasoning trace，也要让用户感知这是一个 Agent 在思考，而不是页面在展示。
