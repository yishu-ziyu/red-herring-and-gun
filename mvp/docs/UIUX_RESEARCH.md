# UI/UX 研究：向顶级产品学习交互模式

> 研究对象：Flowith (flowith.io) / Linear (linear.app) / Obsidian Canvas
> 研究时间：2026-05-21
> 研究者：溯证 Agent 设计团队

---

## 一、Flowith — 空间化 AI 工作区的鼻祖

### 观察到的核心模式

**1. 卡片式画布 (Card-based Canvas)**
- 内容以浮动卡片形式存在于无限画布上
- 卡片大小不一，根据内容重要性自然形成视觉层级
- 卡片之间有微弱的连接线或空间分组暗示关系

**2. 轻量顶部栏 + 局部控制**
- 没有沉重的页面标题区
- 控制元素（模式切换、工具按钮）直接嵌入画布或作为浮动条
- "Composer" 等入口作为小 pill 形状漂浮在右上角

**3. 浅色主题 + 高对比度内容卡片**
- Flowith 使用浅色背景，但卡片使用白色 + 阴影形成层级
- 我们的深色主题版本应反过来：深色背景 + 发光边框/阴影形成层级

### 对溯证 Agent 的借鉴

- ✅ 保留 "节点即内容" 的卡片化思维
- ✅ 右侧/顶部浮动命令入口（对应我们的 Agent 触发面板）
- ✅ 画布是主人，侧边栏是仆人（不能反客为主）
- ⚠️ Flowith 的自动展开可能导致信息过载，我们必须坚持"用户选择节点后才展开"

## 二、Linear — 深色主题与信息密度的极致

### 观察到的核心模式

**1. 边框即层级 (Border as Hierarchy)**
- 面板之间使用 `rgba(255,255,255,0.08~0.12)` 的极淡边框
- 不依赖大面积色块区分区域，而是用微妙的边框 + 背景透明度差异
- 信息密度极高但完全不拥挤

**2. 玻璃拟态的精确控制**
- `backdrop-filter: blur(16~24px)`
- 背景色使用 `rgba(10,10,10,0.7~0.85)` 而非纯黑
- 阴影非常克制：只在大面积浮动面板使用 `0 24px 80px rgba(0,0,0,0.3)`

**3. 侧边栏导航模式**
- 左侧窄 rail (50-60px) + 可展开的侧边面板
- Rail 图标简洁，hover 状态用品牌色填充
- 当前选中状态用背景色块明确标示

**4. 状态色的克制使用**
- 主界面几乎全是灰度
- 状态色（绿/黄/红）只出现在 8px 左右的小圆点或细条中
- 从不让状态色占据大面积区域

**5. 字体层级**
- 大标题：28-32px，粗体，用于核心页面标题
- 面板标题：13-14px，中等粗细，全大写或小型大写
- 正文：14px，常规粗细，行高 1.5
- 标签/元数据：11-12px，粗体，灰色

### 与现有 styles.css 的对比

我们的 styles.css 已经在很大程度上学习了 Linear：
- ✅ 边框透明度 `rgba(243,240,233,0.12)` — 与 Linear 一致
- ✅ `backdrop-filter: blur(26px)` — 接近 Linear 的值
- ✅ 面板背景 `rgba(14,15,13,0.72)` — 正确的玻璃拟态范围
- ✅ 左侧 rail 设计 — 50px 宽度，圆角，图标模式

需要改进的地方：
- ⚠️ 节点目前使用强渐变背景，Linear 风格更偏向纯色 + 微妙边框
- ⚠️ 状态色（绿/橙/红/紫）在节点上面积过大，应缩小到左侧条或顶部条
- ⚠️ Inspector 标题 24px 过大，应降至 18-20px

## 三、Obsidian Canvas — 知识图谱的自由度

### 观察到的核心模式

**1. 内容卡片 + 曲线连接**
- 卡片是内容的一等公民，可以包含文本、图片、甚至嵌套笔记
- 连接线是柔和的贝塞尔曲线，不是直线
- 选中卡片时，相关连接高亮，其他内容淡化

**2. 缩放与平移**
- 鼠标滚轮缩放画布
- 空白处拖拽平移
- 双击或点击按钮聚焦到特定卡片

**3. 分组与对齐**
- 卡片可以编组（视觉上用背景色块包裹）
- 对齐线帮助用户手动排列
- 自动排列为网格或层级

### 对溯证 Agent 的借鉴

- ✅ 曲线连接（已有实现）
- ✅ 选中聚焦模式（高亮路径 + 淡化无关节点）
- ⚠️ 缩放平移对于 MVP 可能过度，暂时不做（会增加复杂度）
- ⚠️ 编组功能可以后期加入，MVP 不做

## 四、综合设计决策

### 保留（styles.css 已很好实现）

| 元素 | 决策 | 理由 |
|------|------|------|
| 深色舞台背景 | 保留 | Linear 验证的深色工作区模式，适合长时间专注 |
| 左侧 rail | 保留 | 50px 宽度合适，图标模式清晰 |
| 玻璃拟态面板 | 保留 | blur(26px) + rgba(14,15,13,0.72) 已很精致 |
| 2.5D 节点厚度 | 保留 | 独特视觉签名，区别于普通流程图 |
| 黄色层级环 | 保留 | 独特的层级表达，但考虑降低不透明度 |

### 改进（基于研究）

| 元素 | 当前 | 改进方向 | 参考 |
|------|------|---------|------|
| 节点状态色 | 整个节点背景变色 | 左侧 4px 竖条 + 顶部小圆点表示状态 | Linear 的状态指示 |
| 节点渐变 | 强蓝渐变 | 降低饱和度，更接近纯色 + 微弱发光 | Linear 的克制 |
| Inspector 标题 | 24px 过大 | 降至 18px，增加字重 | Linear 的信息密度 |
| Claim 输入框 | 较大文本区域 | 启动后缩小为顶部 bar，释放空间 | Flowith 的浮动命令 |
| Agent 触发 | 固定 4 种模式 | 改为上下文敏感的"建议 + 确认"模式 | grill-me 决策 #12 |
| 连接线在聚焦时 | 仅高亮相关边 | 无关节点 opacity 降至 0.3 | Obsidian Canvas |

### 删除（前两版失败教训）

| 元素 | 删除理由 |
|------|---------|
| `revealStage` 线性推进 | 用户无法自由探索，违背"节点选择"核心交互 |
| `advanceReasoning` 按钮 | 强制线性流程，必须改为"点击节点 → 展开选项 → 确认" |
| 固定的 4 种 Agent 模式 | 太死板，应改为 Agent 根据节点类型动态建议 |
| 底部结论 dock 的固定位置 | 可以改为 Inspector 内的可折叠区域 |

## 五、MVP v3 视觉规范更新

### 节点状态指示（新）

```css
/* 状态色只出现在左侧竖条 */
.canvas-node::before {
  content: "";
  position: absolute;
  left: 0;
  top: 12px;
  bottom: 12px;
  width: 3px;
  border-radius: 999px;
  background: var(--status-color);
}

/* 节点背景不再随状态大幅变化 */
.canvas-node {
  background:
    linear-gradient(180deg, rgba(255,255,255,0.08), transparent 46%),
    linear-gradient(135deg, rgba(43,110,203,0.6), rgba(79,153,222,0.5));
}
```

### 聚焦模式（新）

```css
/* 非聚焦节点 */
.canvas-node.dimmed {
  opacity: 0.25;
  filter: grayscale(0.6);
}

/* 聚焦路径上的节点 */
.canvas-node.focused {
  box-shadow: 0 0 0 2px rgba(239,228,93,0.4), 0 20px 42px rgba(0,0,0,0.5);
}
```

### 连接线在聚焦模式（新）

```css
.canvas-edges path.dimmed {
  stroke-opacity: 0.08;
}

.canvas-edges path.focused {
  stroke: rgba(121,185,255,0.8);
  stroke-width: 0.5;
  filter: drop-shadow(0 0 3px rgba(121,185,255,0.4));
}
```

## 六、交互模式清单

### 必须实现的交互

1. **节点点击选中** — 同步 Inspector，高亮节点
2. **节点拖拽** — 改变位置，不改变数据结构
3. **Trace 点击** — 高亮相关节点，Inspector 同步
4. **节点选择后 Agent 建议** — 动态生成展开方向，用户一键确认
5. **聚焦模式** — 点击节点后，高亮路径上的节点和边，淡化其他

### MVP 不做的交互

1. 画布缩放/平移（增加复杂度，无明确需求）
2. 节点编组（后期功能）
3. 多选节点（后期功能）
4. 实时协作（超出范围）
5. 键盘快捷键（后期优化）
