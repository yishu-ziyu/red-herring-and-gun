# 红鲱鱼与枪：按评分标准的产品使用教程

> 面向对象：现场演示者、评委体验引导、内部验收
> 本教程按赛道三“AI Agent - 信息真相猎人”评分标准组织。

## 0. 启动产品

如果本地服务已经启动，直接打开：

```text
http://127.0.0.1:5175/
```

如果没有启动：

```bash
npm run dev -- --host 127.0.0.1 --port 5175
```

进入页面后，第一屏应该看到：

- 产品名：红鲱鱼与枪
- 输入框：输入一条疑似谣言或信息
- 两个主入口：`开始分析`、`深度核查`
- 6 个快速体验案例：健康、社会、科技、财经、政治、娱乐
- 国产链路标签：StepFun / MiMo / DeepSeek / 360 智搜
- 本地知识库状态：本地案例、证据线索

## 1. 总体体验路径

建议先用快速 demo，再用深度核查。

### 路径 A：快速分析

1. 打开首页。
2. 在“快速体验”区域选择一个案例，例如：
   - `隔夜菜会致癌，吃了等于吃毒药`
3. 点击卡片底部的 `快速分析`。
4. 系统直接进入结果态。
5. 观察左侧报告：
   - 待核查信息
   - 可信度分数
   - 公众摘要
   - FIRE 置信度驱动迭代
   - 逻辑风险审计
   - 核查过程
   - 来源列表
6. 观察右侧证据图谱：
   - Agent 节点
   - 来源节点
   - 支持/反驳/报告路径
7. 点击报告中的 `[E1]`、`[E2]` 引用标签，右侧图谱应定位到相关节点。

适合演示：

- 30% 准确性
- 20% 结果闭环
- 10% 知识库

### 路径 B：深度核查

1. 回到首页，点击 `重新核查`。
2. 选择同一个案例或输入自定义传言。
3. 点击 `深度核查`。
4. 系统进入 Mission Control 执行态。
5. 观察 4 个 Agent 的执行流程：
   - RumorDetector
   - FactChecker
   - SourceValidator
   - ReportComposer
6. 观察中央 Agent 大卡片的实时输出。
7. 观察底部时间线，确认每个 Agent 从未开始、执行中到完成。
8. 等执行完成后进入结果态。
9. 在结果态底部确认模型链路，例如：
   - `360-chat:360gpt-pro`
   - `step-3.7-flash`
   - 或明确的 fallback 标签

适合演示：

- 10% Agent 能力
- 10% 360 生态联动
- 10% 技术架构

## 2. 按评分项逐项演示

## 2.1 准确性评估：30%

目标：证明产品不是直接给“真/假”，而是通过证据链、反证、来源质量、逻辑风险来控制误判。

推荐案例：

```text
隔夜菜会致癌，吃了等于吃毒药
```

操作步骤：

1. 首页选择健康类 demo。
2. 点击 `快速分析` 或 `深度核查`。
3. 进入结果页后，先读结论卡片。
4. 找到 `FIRE 置信度驱动迭代`。
5. 逐项解释：
   - 来源可靠性：是否有权威来源。
   - 证据完整度：支持证据和反证是否充足。
   - 逻辑一致性：有没有把相关说成因果。
   - 信息时效性：证据是否过旧。
   - 权威匹配度：来源是否适合支撑该类结论。
6. 找到 `逻辑风险审计`。
7. 重点解释：
   - “相关不等于因果”
   - “弱来源放大风险”
   - “确认偏误风险”
8. 指向报告中的“不能推出 / 不能说”部分。

建议话术：

```text
这个产品的准确性不是来自模型直接判断，而是来自四层约束：
第一，先拆解原始传言里的子命题；
第二，同时寻找支持证据和反驳证据；
第三，对来源可信度、时效性、多样性评分；
第四，对因果跳跃和认知偏差做审计。

所以它不会把“隔夜菜含亚硝酸盐”直接推成“隔夜菜等于毒药”，而是会提示证据能支持到什么程度、不能推出什么。
```

评委看到的加分点：

- 不是普通聊天机器人。
- 能说明“为什么不能下强结论”。
- 有反证意识。
- 有来源质量和逻辑风险解释。

## 2.2 场景覆盖评估：20%

目标：证明至少覆盖 3 类谣言，最好展示 6 类。

首页目前提供 6 类：

| 类型 | Demo |
|---|---|
| 健康 | 隔夜菜会致癌，吃了等于吃毒药 |
| 社会 | 某城市地铁即将停运，内部消息 |
| 科技 | 5G信号塔辐射导致周边居民头晕失眠 |
| 财经 | 人民币即将大幅贬值，赶紧换美元 |
| 政治 | 政策/内部文件类谣言 |
| 娱乐 | 明星/截图/匿名爆料类谣言 |

操作步骤：

1. 在首页依次展示 6 个 demo 卡片。
2. 不需要每个都跑完深度核查，建议现场完整跑 2 个，快速切换展示另外 4 个。
3. 优先完整跑：
   - 健康类：展示因果误导。
   - 社会类：展示匿名信源和传播诱导。
   - 科技类：展示概念偷换。
4. 如果时间允许，再展示财经、政治、娱乐卡片，说明覆盖更多类型。

建议话术：

```text
评分要求至少 3 类，我们现在覆盖 6 类：健康、社会、科技、财经、政治、娱乐。
每类不是换一个文案，而是有不同识别重点：
健康类看伪科学和恐惧诉求；
社会类看匿名信源和群体焦虑；
科技类看概念偷换；
财经类看预测煽动；
政治类看政策误读；
娱乐类看截图断章和名誉风险。
```

评委看到的加分点：

- 场景覆盖超过最低要求。
- 每类谣言有不同识别逻辑。
- 不是只靠关键词分类。

## 2.3 结果闭环评估：20%

目标：证明核查不是到报告为止，而是有完整后续动作。

操作步骤：

1. 跑完任意案例，进入结果态。
2. 看页面底部操作栏。
3. 依次演示 4 个动作：
   - `导出报告`
   - `辟谣卡片`
   - `存疑归档`
   - `分享核查`
4. 点击 `导出报告`：
   - 生成 Markdown 报告。
   - 说明适合交给研究、媒体或团队归档。
5. 点击 `辟谣卡片`：
   - 生成面向公众传播的简版结论。
   - 说明适合发群聊或朋友圈。
6. 点击 `存疑归档`：
   - 页面底部会显示存疑数量。
   - 说明证据不足的 case 不会丢失，会进入知识库。
7. 点击 `分享核查`：
   - 如果浏览器支持系统分享，则调用系统分享。
   - 否则复制分享文本。

建议话术：

```text
我们把结果闭环拆成四个动作：
可信的，导出报告；
明显误导的，生成辟谣卡片；
证据不足的，存疑归档；
需要传播的，分享核查。

这样产品不只是“告诉你真假”，而是把核查结果转化成用户下一步能做的行动。
```

评委看到的加分点：

- 闭环动作完整。
- 面向真实传播场景。
- 证据不足也有归档路径。

## 2.4 技术架构评估：10%

目标：证明有 Agent 记忆机制、知识库机制、证据复用、多 Agent 联动。

操作步骤：

1. 在首页观察 `本地知识库状态`。
2. 跑完一个案例后，点击 `重新核查` 回到首页。
3. 再输入类似传言，例如：

```text
隔夜菜里面全是亚硝酸盐，长期吃一定会得癌症
```

4. 输入后观察是否出现 `相似历史案例`。
5. 点击相似历史案例。
6. 说明系统会复用历史 case 的证据线索和判断边界。
7. 再进入深度核查，展示 Agent Team：
   - RumorDetector 识别谣言特征。
   - FactChecker 做事实与反证核查。
   - SourceValidator 做来源验证。
   - ReportComposer 生成报告和闭环动作。

建议话术：

```text
这里的知识库不是简单收藏夹。
每次核查完成后，系统会把 claim、谣言类型、报告、证据线索、可信度、Agent steps 保存为本地 case。
下次遇到相似传言时，会先召回历史案例，复用已有证据和判断边界。

同时多个 Agent 不是并列聊天，而是有明确分工和交接：
识别、事实核查、信源验证、报告生成。
```

评委看到的加分点：

- 有持久化案例。
- 有相似 case 召回。
- 有证据线索复用。
- 有多 Agent 分工和状态流转。

## 2.5 Agent 能力评估：10%

目标：证明使用国产大模型作为 Agent 基础能力，并且效果不是只依赖国外模型。

操作步骤：

1. 首页指向 `国产链路` 标签。
2. 选择任意 demo，点击 `深度核查`。
3. 在 Mission Control 中观察 Agent 输出。
4. 完成后进入结果态。
5. 看底部模型摘要。
6. 如果显示：
   - `360-chat:360gpt-pro`
   - `step-3.7-flash`
   - `mimo`
   - `deepseek`
   说明命中了国产模型链路。
7. 如果显示 `demo-fallback`，说明当前网络或 key 没命中真实模型，需要检查环境变量。

建议话术：

```text
我们的 Agent 基础模型优先走国产链路：
360GPT、StepFun、MiMo、DeepSeek。
国外模型不是主依赖，只作为极端情况下的兜底。

现场演示时，结果态底部会展示真实 provider 名称，避免把 demo fallback 伪装成模型能力。
```

评委看到的加分点：

- 国产模型路径明确。
- UI 可见模型来源。
- fallback 透明，不伪装。

## 2.6 360 生态联动：10%

目标：证明产品真正联动 360 生态，至少满足一个产品联动。

操作步骤：

1. 首页说明当前联动：
   - 360GPT：Agent 推理和报告生成。
   - 360 AI Search：搜索增强和来源检索。
2. 选择健康类或社会类传言，点击 `深度核查`。
3. 等结果完成。
4. 在结果页观察：
   - 可复核来源。
   - 模型摘要。
   - 证据图谱。
5. 如果需要技术验收，可在终端执行：

```bash
curl -s -X POST http://127.0.0.1:5175/api/search/360 \
  -H 'Content-Type: application/json' \
  -d '{"query":"隔夜菜会致癌，吃了等于吃毒药","refProm":"aiso-max"}'
```

合格结果：

```text
model: "360-ai-search:360gpt-pro"
```

建议话术：

```text
360 生态在这里不是贴 logo，而是进入了核查链路：
360 AI Search 负责找来源、做搜索增强；
360GPT 进入 Agent 推理链路；
最终报告会把模型来源和证据来源展示出来。
```

评委看到的加分点：

- 360 AI Search 有真实 API 路径。
- 360GPT 有真实 Agent 推理路径。
- UI 和技术验收都能证明联动。

## 3. 推荐现场演示顺序

总时长：5 分钟。

### 第 1 分钟：定位

1. 打开首页。
2. 说一句：

```text
这是一个中文互联网传言的证据链 Agent Team。
它不只判断真假，而是拆解断言、寻找支持和反证、审计来源与逻辑风险，最后给出可传播、可归档、可复用的核查结果。
```

### 第 2 分钟：准确性

1. 选择健康类 demo。
2. 点击 `快速分析`。
3. 展示结论、FIRE、逻辑风险审计。

### 第 3 分钟：Agent Team

1. 点击 `重新核查`。
2. 再选择同一案例。
3. 点击 `深度核查`。
4. 展示 Mission Control 的 4 个 Agent 执行过程。

### 第 4 分钟：结果闭环

1. 进入结果态。
2. 演示：
   - 辟谣卡片
   - 存疑归档
   - 分享核查

### 第 5 分钟：评分收束

1. 回首页展示 6 类 demo。
2. 指向知识库状态。
3. 指向国产链路和 360 联动。
4. 结尾话术：

```text
这套产品对应评分项是完整闭环：
准确性靠证据链和偏差审计；
覆盖度有 6 类中文谣言；
结果闭环有报告、卡片、归档、分享；
技术架构有 Agent Team 和本地知识库；
Agent 能力优先国产模型；
360 生态接入 360GPT 和 360 AI Search。
```

## 5. 失败时的应对

### 模型 API 没通

说法：

```text
当前环境走 demo fallback，但 fallback 会明确标记，不会伪装成真实模型。
真实链路已经配置 360GPT / StepFun / MiMo / DeepSeek，部署时只需要补 secret。
```

### 360 搜索没通

说法：

```text
如果 360 AI Search 不可用，会回退到 360 mwebsearch，再不行才使用 demo fallback。
评分演示时优先用已充值 key 的本地环境。
```

### 证据不足

不要说产品失败。正确说法：

```text
证据不足是事实核查里的正常状态。
我们的产品会把它转成闭环动作：存疑归档、继续补证、降低结论强度，而不是硬判真假。
```
